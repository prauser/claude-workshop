# Agent Teams vs Orchestrator + Sub-agents

Claude Code에서 복수 에이전트를 활용하는 두 가지 패턴의 비교 정리.

## 아키텍처 차이

| | Orchestrator + Sub-agent | Agent Teams |
|---|---|---|
| 구조 | 계층적 (메인 → 서브 → 결과 반환) | 피어-투-피어 (팀원끼리 직접 대화) |
| 세션 | 하나의 세션 안에서 서브프로세스 | 각 팀원이 독립 세션 (별도 컨텍스트 윈도우) |
| 통신 | 서브 → 메인으로만 결과 보고 | 팀원끼리 메일박스로 직접 메시지 교환 |
| 조율 | 메인 에이전트가 모든 분배/판단 | 공유 태스크 리스트에서 자율적으로 작업 claim |
| 비용 | 결과 요약만 컨텍스트에 들어옴 → 저렴 | 각 팀원이 풀 컨텍스트 → 비용 높음 |

## 핵심 구분 기준

- **Sub-agent**: "실행 결과가 가치인 작업" — 시키고 결과 받기
- **Agent Teams**: "합의 도출 과정 자체가 가치인 문제" — 팀원끼리 토론하고 결론 보고

구현/리뷰/테스트처럼 정답이 비교적 명확한 작업은 서브에이전트로 충분하다.
Teams가 의미 있는 건 **뭐가 맞는지 모르는 상황**, 즉 다채로운 분석이나 트레이드오프 토의가 필요한 경우다.

## Sub-agent가 적합한 케이스 (~85%)

### 기능 구현
명확한 스펙 → 구현 → 리뷰 → 테스트. 순차 파이프라인이면 충분.

### 버그 수정 (원인이 짐작 가능할 때)
탐색 에이전트가 찾고 → 구현 에이전트가 고치고 → 끝.

### 리팩토링
정답이 상대적으로 명확. 병렬 worktree로 각자 파일 수정.

### 테스트 작성, 문서 업데이트, 마이그레이션
전부 "실행 결과"가 가치인 작업.

## Agent Teams가 적합한 케이스 (~15%)

### 1. 장애 원인 분석 — 원인을 모를 때
- 팀원 A: DB 쿼리 플랜 분석
- 팀원 B: 네트워크/커넥션 풀 조사
- 팀원 C: 최근 배포 diff 검토
- A가 "쿼리는 정상" → B가 "그러면 커넥션 풀 쪽이 수상" → 실시간으로 좁혀감

### 2. 설계 의사결정 — 트레이드오프가 복잡할 때
- 예: WebSocket vs SSE vs 폴링
- 각 관점(성능, 인프라 비용, 클라이언트 복잡도)에서 동시에 논증하고 반박

### 3. 대규모 마이그레이션 전략 수립
- 예: Python 2→3, REST→GraphQL
- 영향 범위 분석 + 리스크 평가 + 단계별 계획을 각자 조사 후 합의

### 4. 보안 감사 — 다각도 동시 검토
- 팀원별로 인증, 인젝션, 권한 상승 등 다른 공격 벡터를 동시 탐색
- 크로스 발견: "이쪽 취약점 + 네 쪽 권한 로직 = 체인 공격 가능"

## 비용 현실

Agent Teams는 팀원 수 × 풀 컨텍스트만큼 토큰을 소비한다.
30분 고민해서 결론낼 설계 문제를 5분에 끝낸다면 가치가 있지만,
혼자 생각해도 되는 수준이면 과잉이다.

## 활성화 방법

Agent Teams는 실험 기능이므로 환경변수로 활성화 필요:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```
