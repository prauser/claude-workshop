# Code Review and Quality

## 개요

품질 게이트가 있는 다차원 코드 리뷰. 모든 변경사항은 예외 없이 머지 전에 리뷰를 받습니다. 리뷰는 다섯 가지 축을 다룹니다: 정확성, 가독성, 아키텍처, 보안, 성능.

**승인 기준:** 코드가 완벽하지 않더라도 전반적인 코드 상태를 분명히 개선할 때 변경사항을 승인하세요. 완벽한 코드는 존재하지 않습니다 — 목표는 지속적인 개선입니다. 자신이 작성했을 방식과 정확히 같지 않다는 이유로 변경사항을 차단하지 마세요. 코드베이스를 개선하고 프로젝트의 컨벤션을 따른다면, 승인하세요.

## 언제 사용할까

- 어떤 PR 또는 변경사항을 머지하기 전에
- 기능 구현을 완료한 후
- 다른 에이전트나 모델이 생성한 코드를 평가해야 할 때
- 기존 코드를 리팩토링할 때
- 버그 수정 후 (수정사항과 회귀 테스트 모두 리뷰)

## 다섯 가지 축 리뷰

모든 리뷰는 이 차원들에 걸쳐 코드를 평가합니다:

### 1. 정확성

코드가 주장하는 것을 실제로 하는가?

- 명세 또는 작업 요구사항과 일치하는가?
- 엣지 케이스가 처리되는가 (null, empty, 경계값)?
- 에러 경로가 처리되는가 (해피 패스만이 아닌)?
- 모든 테스트를 통과하는가? 테스트가 실제로 올바른 것을 테스트하는가?
- off-by-one 오류, 경쟁 조건, 또는 상태 불일치가 있는가?

### 2. 가독성 및 단순성

다른 엔지니어(또는 에이전트)가 저자의 설명 없이 이 코드를 이해할 수 있는가?

- 이름이 프로젝트 컨벤션과 일관적이고 설명적인가? (컨텍스트 없는 `temp`, `data`, `result` 없음)
- 제어 흐름이 직관적인가 (중첩된 삼항 연산자, 깊은 콜백 피하기)?
- 코드가 논리적으로 구성되어 있는가 (관련 코드가 묶이고, 명확한 모듈 경계)?
- 단순화되어야 하는 "영리한" 트릭이 있는가?
- **더 적은 줄로 할 수 있는가?** (100줄로 충분한 곳에 1000줄은 실패)
- **추상화가 그 복잡성에 값어치를 하는가?** (세 번째 사용 케이스까지 일반화하지 마세요)
- 명백하지 않은 의도를 명확히 하기 위해 주석이 도움이 되는가? (하지만 명백한 코드에 주석을 달지 마세요.)
- 죽은 코드 잔재물이 있는가: no-op 변수 (`_unused`), 하위 호환 심, 또는 `// removed` 주석?

### 3. 아키텍처

변경사항이 시스템의 설계에 맞는가?

- 기존 패턴을 따르는가, 아니면 새로운 것을 도입하는가? 새로운 것이라면 정당화되는가?
- 깨끗한 모듈 경계를 유지하는가?
- 공유되어야 할 코드 중복이 있는가?
- 의존성이 올바른 방향으로 흐르는가 (순환 의존성 없음)?
- 추상화 수준이 적절한가 (과도하게 엔지니어링되지 않음, 너무 결합되지 않음)?

### 4. 보안

자세한 보안 지침은 `security-and-hardening`을 참조하세요. 변경사항이 취약점을 도입하는가?

- 사용자 입력이 유효성 검사되고 정제되는가?
- 시크릿이 코드, 로그, 버전 관리에서 제외되는가?
- 필요한 곳에 인증/권한 부여가 확인되는가?
- SQL 쿼리가 파라미터화되어 있는가 (문자열 연결 없음)?
- XSS를 방지하기 위해 출력이 인코딩되는가?
- 의존성이 알려진 취약점이 없는 신뢰할 수 있는 소스에서 온 것인가?
- 외부 소스 (API, 로그, 사용자 콘텐츠, 설정 파일)의 데이터가 신뢰할 수 없는 것으로 취급되는가?
- 외부 데이터 흐름이 로직이나 렌더링에 사용되기 전에 시스템 경계에서 유효성 검사되는가?

### 5. 성능

자세한 프로파일링 및 최적화는 `performance-optimization`을 참조하세요. 변경사항이 성능 문제를 도입하는가?

- N+1 쿼리 패턴이 있는가?
- 무한 루프 또는 제약 없는 데이터 페칭이 있는가?
- async여야 할 동기 작업이 있는가?
- UI 컴포넌트에 불필요한 리렌더링이 있는가?
- 목록 엔드포인트에 페이지네이션이 없는가?
- 핫 경로에서 생성되는 큰 객체가 있는가?

## 변경 크기

작고 집중된 변경사항은 리뷰하기 쉽고, 더 빠르게 머지되며, 배포하기 더 안전합니다. 이 크기를 목표로 하세요:

```
~100줄 변경   → 좋음. 한 번에 리뷰 가능.
~300줄 변경   → 단일 논리적 변경이라면 수용 가능.
~1000줄 변경  → 너무 큼. 분리하세요.
```

**"하나의 변경"이란:** 하나의 것을 다루고, 관련 테스트를 포함하며, 제출 후 시스템이 기능하는 상태를 유지하는 단일하고 자체 포함된 수정. 기능의 한 부분 — 전체 기능이 아닙니다.

**변경이 너무 클 때 분리 전략:**

| 전략 | 방법 | 언제 |
|----------|-----|------|
| **Stack** | 작은 변경을 제출하고, 그 위에 다음 것을 시작 | 순차적 의존성 |
| **파일 그룹별** | 다른 리뷰어가 필요한 그룹에 대해 변경 분리 | 교차 관심사 |
| **수평적** | 먼저 공유 코드/스텁을 생성하고, 그 다음 소비자 | 계층화된 아키텍처 |
| **vertical slice** | 기능의 더 작은 풀스택 slice로 분리 | 기능 작업 |

**큰 변경이 허용되는 경우:** 리뷰어가 모든 줄이 아닌 의도만 확인하면 되는 완전한 파일 삭제 및 자동화된 리팩토링.

**리팩토링과 기능 작업을 분리하세요.** 기존 코드를 리팩토링하고 새로운 동작을 추가하는 변경은 두 개의 변경입니다 — 따로 제출하세요. 작은 정리 (변수 이름 변경)는 리뷰어 재량으로 포함할 수 있습니다.

## 변경 설명

모든 변경사항은 버전 관리 히스토리에서 독립적으로 설명할 수 있는 설명이 필요합니다.

**첫 번째 줄:** 짧고, 명령형, 독립적. "Delete the FizzBuzz RPC" (not "Deleting the FizzBuzz RPC"). 누군가 히스토리를 검색할 때 diff를 읽지 않고도 변경을 이해할 수 있을 만큼 충분히 정보적이어야 합니다.

**본문:** 무엇이 변경되고 왜 변경되는지. 코드 자체에서 보이지 않는 컨텍스트, 결정, 이유를 포함하세요. 관련 있는 경우 버그 번호, 벤치마크 결과, 또는 설계 문서를 링크하세요. 접근 방식의 단점이 있다면 인정하세요.

**안티패턴:** "Fix bug," "Fix build," "Add patch," "Moving code from A to B," "Phase 1," "Add convenience functions."

## 리뷰 프로세스

### 1단계: 컨텍스트 이해

코드를 보기 전에 의도를 이해하세요:

```
- 이 변경사항은 무엇을 달성하려는가?
- 어떤 명세 또는 작업을 구현하는가?
- 예상되는 동작 변화는 무엇인가?
```

### 2단계: 테스트 먼저 리뷰

테스트는 의도와 범위를 드러냅니다:

```
- 변경에 대한 테스트가 있는가?
- 동작을 테스트하는가 (구현 세부사항이 아닌)?
- 엣지 케이스가 다루어지는가?
- 테스트에 설명적인 이름이 있는가?
- 코드가 변경된다면 테스트가 회귀를 잡을 수 있는가?
```

### 3단계: 구현 리뷰

다섯 가지 축을 염두에 두고 코드를 살펴보세요:

```
변경된 각 파일에 대해:
1. 정확성: 이 코드가 테스트가 말하는 것을 하는가?
2. 가독성: 도움 없이 이해할 수 있는가?
3. 아키텍처: 시스템에 맞는가?
4. 보안: 취약점이 있는가?
5. 성능: 병목이 있는가?
```

### 4단계: 발견사항 분류

작성자가 필수와 선택을 알 수 있도록 모든 코멘트에 심각도를 표시하세요:

| 접두사 | 의미 | 작성자 조치 |
|--------|---------|---------------|
| *(접두사 없음)* | 필수 변경 | 머지 전에 반드시 해결 |
| **Critical:** | 머지 차단 | 보안 취약점, 데이터 손실, 깨진 기능 |
| **Nit:** | 사소함, 선택적 | 작성자가 무시 가능 — 형식, 스타일 선호도 |
| **Optional:** / **Consider:** | 제안 | 고려할 가치가 있지만 필수 아님 |
| **FYI** | 정보 제공용 | 조치 필요 없음 — 미래 참고를 위한 컨텍스트 |

이는 작성자가 모든 피드백을 의무적인 것으로 취급하고 선택적인 제안에 시간을 낭비하는 것을 방지합니다.

### 5단계: 검증 스토리 확인

작성자의 검증 스토리를 확인하세요:

```
- 어떤 테스트가 실행되었는가?
- 빌드가 통과했는가?
- 변경이 수동으로 테스트되었는가?
- UI 변경에 대한 스크린샷이 있는가?
- 변경 전후 비교가 있는가?
```

## 멀티 모델 리뷰 패턴

다른 리뷰 관점을 위해 다른 모델을 사용하세요:

```
모델 A가 코드를 작성
    │
    ▼
모델 B가 정확성과 아키텍처를 리뷰
    │
    ▼
모델 A가 피드백을 반영
    │
    ▼
사람이 최종 결정
```

이는 단일 모델이 놓칠 수 있는 문제를 잡습니다 — 다른 모델은 다른 맹점을 가집니다.

**리뷰 에이전트를 위한 예시 프롬프트:**
```
이 코드 변경을 정확성, 보안, 그리고
우리 프로젝트 컨벤션 준수 여부에 대해 리뷰하세요. 명세는 [X]입니다. 변경은 [Y]여야 합니다.
모든 문제를 Critical, Important, 또는 Suggestion으로 표시하세요.
```

## 죽은 코드 위생

리팩토링 또는 구현 변경 후, 고아 코드를 확인하세요:

1. 이제 도달할 수 없거나 사용되지 않는 코드를 파악하세요
2. 명시적으로 나열하세요
3. **삭제 전에 질문하세요:** "이제 사용되지 않는 요소들을 제거해도 될까요: [목록]?"

죽은 코드를 방치하지 마세요 — 미래의 독자와 에이전트를 혼란스럽게 합니다. 하지만 확실하지 않은 것을 조용히 삭제하지도 마세요. 의심스러울 때는 질문하세요.

```
죽은 코드 확인:
- formatLegacyDate() in src/utils/date.ts — formatDate()로 대체됨
- OldTaskCard component in src/components/ — TaskCard로 대체됨
- LEGACY_API_URL constant in src/config.ts — 남은 참조 없음
→ 이것들을 제거해도 안전한가요?
```

## 리뷰 속도

느린 리뷰는 전체 팀을 차단합니다. 리뷰를 위해 컨텍스트를 전환하는 비용은 다른 사람들에게 부과되는 대기 비용보다 낮습니다.

- **영업일 하루 내에 응답** — 이것은 최대값이지 목표값이 아닙니다
- **이상적인 주기:** 집중 코딩 중이 아니라면 리뷰 요청이 도착하고 얼마 지나지 않아 응답하세요. 일반적인 변경은 하루 내에 여러 리뷰 라운드를 완료해야 합니다
- **빠른 최종 승인보다 빠른 개별 응답을 우선시하세요.** 빠른 피드백은 여러 라운드가 필요하더라도 좌절을 줄입니다
- **큰 변경:** 하나의 방대한 변경셋을 리뷰하는 대신 작성자에게 분리를 요청하세요

## 이견 처리

리뷰 분쟁을 해결할 때, 이 계층을 적용하세요:

1. **기술적 사실과 데이터**가 의견과 선호도를 이깁니다
2. **스타일 가이드**는 스타일 문제의 절대적인 권위입니다
3. **소프트웨어 설계**는 개인 선호도가 아닌 엔지니어링 원칙으로 평가되어야 합니다
4. **코드베이스 일관성**은 전반적인 상태를 저하시키지 않는다면 수용 가능합니다

**"나중에 정리하겠습니다"를 수락하지 마세요.** 경험상 지연된 정리는 거의 일어나지 않습니다. 진정한 긴급 상황이 아니라면 제출 전에 정리를 요구하세요. 이번 변경에서 주변의 문제를 해결할 수 없다면, 자기 할당으로 버그를 제출하도록 요구하세요.

## 리뷰의 정직함

자신, 다른 에이전트, 또는 사람이 작성한 코드를 리뷰할 때:

- **무작정 승인하지 마세요.** 리뷰의 증거 없는 "LGTM"은 아무에게도 도움이 되지 않습니다.
- **실제 문제를 부드럽게 표현하지 마세요.** "이것은 사소한 우려일 수 있습니다"라고 말하면서 사실은 프로덕션에 영향을 줄 버그라면 그것은 부정직한 것입니다.
- **가능할 때 문제를 정량화하세요.** "이 N+1 쿼리는 목록의 항목당 ~50ms를 추가할 것입니다"가 "이것은 느릴 수 있습니다"보다 낫습니다.
- **명확한 문제가 있는 접근 방식에 반대하세요.** 아첨은 리뷰의 실패 모드입니다. 구현에 문제가 있다면 직접 말하고 대안을 제안하세요.
- **오버라이드를 우아하게 수용하세요.** 작성자가 완전한 컨텍스트를 가지고 동의하지 않는다면, 그들의 판단에 따르세요. 사람이 아닌 코드에 대해 코멘트하세요 — 개인적 비판을 코드 자체에 초점을 맞추도록 재구성하세요.

## 의존성 규율

코드 리뷰의 일부는 의존성 리뷰입니다:

**의존성을 추가하기 전에:**
1. 기존 스택이 이것을 해결하는가? (종종 그렇습니다.)
2. 의존성의 크기는 얼마나 되는가? (번들 영향 확인.)
3. 활발하게 유지보수되고 있는가? (마지막 커밋, 열린 이슈 확인.)
4. 알려진 취약점이 있는가? (`npm audit`)
5. 라이선스는 무엇인가? (프로젝트와 호환되어야 합니다.)

**규칙:** 새로운 의존성보다 표준 라이브러리와 기존 유틸리티를 선호하세요. 모든 의존성은 부채입니다.

## 리뷰 체크리스트

```markdown
## Review: [PR/변경 제목]

### 컨텍스트
- [ ] 이 변경이 무엇을 하고 왜 하는지 이해함

### 정확성
- [ ] 변경이 명세/작업 요구사항과 일치함
- [ ] 엣지 케이스 처리됨
- [ ] 에러 경로 처리됨
- [ ] 테스트가 변경을 충분히 다룸

### 가독성
- [ ] 이름이 명확하고 일관적임
- [ ] 로직이 직관적임
- [ ] 불필요한 복잡성 없음

### 아키텍처
- [ ] 기존 패턴을 따름
- [ ] 불필요한 결합 또는 의존성 없음
- [ ] 적절한 추상화 수준

### 보안
- [ ] 코드에 시크릿 없음
- [ ] 경계에서 입력 유효성 검사됨
- [ ] 주입 취약점 없음
- [ ] 인증 검사가 존재함
- [ ] 외부 데이터 소스가 신뢰할 수 없는 것으로 취급됨

### 성능
- [ ] N+1 패턴 없음
- [ ] 제약 없는 작업 없음
- [ ] 목록 엔드포인트에 페이지네이션

### 검증
- [ ] 테스트 통과
- [ ] 빌드 성공
- [ ] 수동 검증 완료 (해당되는 경우)

### 평결
- [ ] **승인** — 머지 준비 완료
- [ ] **변경 요청** — 문제를 반드시 해결해야 함
```

## 흔한 합리화

| 합리화 | 현실 |
|---|---|
| "동작하면 충분합니다" | 읽을 수 없고, 안전하지 않고, 아키텍처적으로 잘못된 동작하는 코드는 복리로 쌓이는 부채를 만듭니다. |
| "내가 작성했으니 맞을 것입니다" | 저자는 자신의 가정에 눈이 됩니다. 모든 변경은 다른 눈의 혜택을 받습니다. |
| "나중에 정리하겠습니다" | 나중은 오지 않습니다. 리뷰가 품질 게이트입니다 — 사용하세요. 머지 후가 아닌 전에 정리를 요구하세요. |
| "AI 생성 코드는 아마 괜찮을 것입니다" | AI 코드는 더 적은 것이 아닌 더 많은 검토가 필요합니다. 그것은 자신감 있고 그럴듯하지만, 틀릴 때도 마찬가지입니다. |
| "테스트가 통과하니 좋습니다" | 테스트는 필요하지만 충분하지 않습니다. 아키텍처 문제, 보안 문제, 또는 가독성 우려를 잡지 않습니다. |

## 위험 신호

- 어떤 리뷰도 없이 머지된 PR
- 테스트가 통과하는지만 확인하는 리뷰 (다른 축 무시)
- 실제 리뷰의 증거 없는 "LGTM"
- 보안 중심 리뷰 없는 보안 민감한 변경
- "제대로 리뷰하기엔 너무 큰" 큰 PR (분리하세요)
- 버그 수정 PR에 회귀 테스트 없음
- 심각도 표시 없는 리뷰 코멘트 — 필수 대 선택이 불명확함
- "나중에 수정하겠습니다" 수락 — 절대 일어나지 않습니다

- [ ] 빌드 성공
