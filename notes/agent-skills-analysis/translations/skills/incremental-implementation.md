# Incremental Implementation

## Overview

얇은 vertical slice로 구축하라 — 하나의 조각을 구현하고, 테스트하고, 확인한 다음 확장한다. 한 번에 전체 기능을 구현하는 것을 피한다. 각 단계는 시스템을 작동하고 테스트 가능한 상태로 남겨야 한다. 이것이 대규모 기능을 관리 가능하게 만드는 실행 규율이다.

## When to Use

- 멀티 파일 변경 구현
- 작업 분해에서 새 기능 구축
- 기존 코드 리팩터링
- 테스트 전에 100줄 이상을 작성하고 싶은 유혹이 들 때

**사용하지 않을 때:** 범위가 이미 최소인 단일 파일, 단일 함수 변경.

## The Increment Cycle

```
┌──────────────────────────────────────┐
│                                      │
│   구현 ──→ 테스트 ──→ 확인 ──┐       │
│     ▲                        │       │
│     └───── 커밋 ◄─────────────┘       │
│            │                         │
│            ▼                         │
│        다음 슬라이스                  │
│                                      │
└──────────────────────────────────────┘
```

각 슬라이스에 대해:

1. **구현** — 가장 작은 완전한 기능 조각
2. **테스트** — 테스트 스위트 실행 (또는 없다면 테스트 작성)
3. **확인** — 슬라이스가 예상대로 작동하는지 확인 (테스트 통과, 빌드 성공, 수동 확인)
4. **커밋** — 설명적인 메시지와 함께 진행 사항 저장 (atomic commit 지침은 `git-workflow-and-versioning` 참조)
5. **다음 슬라이스로 이동** — 앞으로 나아가기, 재시작하지 않기

## Slicing Strategies

### Vertical Slices (권장)

스택을 통해 하나의 완전한 경로를 구축한다:

```
슬라이스 1: 태스크 생성 (DB + API + 기본 UI)
    → 테스트 통과, 사용자가 UI를 통해 태스크를 생성할 수 있음

슬라이스 2: 태스크 목록 (쿼리 + API + UI)
    → 테스트 통과, 사용자가 자신의 태스크를 볼 수 있음

슬라이스 3: 태스크 편집 (업데이트 + API + UI)
    → 테스트 통과, 사용자가 태스크를 수정할 수 있음

슬라이스 4: 태스크 삭제 (삭제 + API + UI + 확인)
    → 테스트 통과, 전체 CRUD 완성
```

각 슬라이스는 end-to-end로 작동하는 기능을 제공한다.

### Contract-First Slicing

백엔드와 프론트엔드가 병렬로 개발해야 할 때:

```
슬라이스 0: API 계약 정의 (타입, 인터페이스, OpenAPI spec)
슬라이스 1a: 계약에 맞게 백엔드 구현 + API 테스트
슬라이스 1b: 계약에 맞는 mock 데이터로 프론트엔드 구현
슬라이스 2: 통합 및 end-to-end 테스트
```

### Risk-First Slicing

가장 위험하거나 불확실한 조각을 먼저 다룬다:

```
슬라이스 1: WebSocket 연결이 작동하는지 증명 (가장 높은 위험)
슬라이스 2: 증명된 연결 위에 실시간 태스크 업데이트 구축
슬라이스 3: 오프라인 지원 및 재연결 추가
```

슬라이스 1이 실패하면, 슬라이스 2와 3에 투자하기 전에 발견한다.

## Implementation Rules

### Rule 0: Simplicity First

어떤 코드를 작성하기 전에 물어라: "작동할 수 있는 가장 단순한 것은 무엇인가?"

코드를 작성한 후 다음 체크에 대해 검토한다:
- 더 적은 줄로 할 수 있는가?
- 이 추상화들이 복잡성의 가치를 하는가?
- 시니어 엔지니어가 이것을 보고 "왜 그냥..."이라고 말하지 않을까?
- 가상의 미래 요구 사항이 아닌 현재 작업을 위해 구축하고 있는가?

```
단순성 체크:
✗ 하나의 알림을 위한 미들웨어 파이프라인이 있는 일반적인 EventBus
✓ 단순한 함수 호출

✗ 두 개의 유사한 컴포넌트를 위한 추상 팩토리 패턴
✓ 공유 유틸리티가 있는 두 개의 간단한 컴포넌트

✗ 세 개의 폼을 위한 설정 기반 폼 빌더
✓ 세 개의 폼 컴포넌트
```

유사한 코드 세 줄이 성급한 추상화보다 낫다. 먼저 naive하고 명백히 올바른 버전을 구현하라. 테스트로 정확성이 증명된 후에만 최적화한다.

### Rule 0.5: Scope Discipline

작업이 요구하는 것만 건드린다.

다음을 하지 않는다:
- 변경 사항 인접한 코드를 "정리"하기
- 수정하지 않는 파일에서 import 리팩터링
- 완전히 이해하지 못한 주석 제거
- "유용해 보인다"는 이유로 spec에 없는 기능 추가
- 단순히 읽고 있는 파일의 문법 현대화

작업 범위 밖에서 개선할 가치가 있는 것을 발견하면, 메모하라 — 수정하지 않는다:

```
발견했지만 건드리지 않는 것:
- src/utils/format.ts에 사용되지 않는 import가 있음 (이 작업과 무관)
- auth 미들웨어가 더 나은 에러 메시지를 사용할 수 있음 (별도 작업)
→ 이것들을 위한 작업을 만들까요?
```

### Rule 1: One Thing at a Time

각 단계는 하나의 논리적인 것을 변경한다. 관심사를 혼합하지 않는다:

**나쁨:** 새 컴포넌트를 추가하고, 기존 것을 리팩터링하며, 빌드 설정을 업데이트하는 하나의 커밋.

**좋음:** 세 개의 별도 커밋 — 각 변경당 하나.

### Rule 2: Keep It Compilable

각 단계 후, 프로젝트는 빌드되어야 하고 기존 테스트는 통과해야 한다. 슬라이스 사이에 코드베이스를 망가진 상태로 두지 않는다.

### Rule 3: Feature Flags for Incomplete Features

기능이 사용자에게 준비되지 않았지만 단계를 병합해야 하는 경우:

```typescript
// 진행 중인 작업을 위한 feature flag
const ENABLE_TASK_SHARING = process.env.FEATURE_TASK_SHARING === 'true';

if (ENABLE_TASK_SHARING) {
  // 새 공유 UI
}
```

이를 통해 미완성 작업을 사용자에게 노출하지 않고 소규모 단계를 main 브랜치에 병합할 수 있다.

### Rule 4: Safe Defaults

새 코드는 기본적으로 안전하고 보수적인 동작을 해야 한다:

```typescript
// 안전함: 기본값이 비활성화, opt-in
export function createTask(data: TaskInput, options?: { notify?: boolean }) {
  const shouldNotify = options?.notify ?? false;
  // ...
}
```

### Rule 5: Rollback-Friendly

각 단계는 독립적으로 되돌릴 수 있어야 한다:

- 추가적인 변경 (새 파일, 새 함수)은 되돌리기 쉬움
- 기존 코드에 대한 수정은 최소화하고 집중적이어야 함
- 데이터베이스 마이그레이션에는 해당 롤백 마이그레이션이 있어야 함
- 하나의 커밋에서 무언가를 삭제하고 같은 커밋에서 교체하는 것을 피하라 — 분리하라

## Working with Agents

에이전트가 점진적으로 구현하도록 지시할 때:

```
"계획에서 Task 3을 구현해 보자.

데이터베이스 스키마 변경과 API 엔드포인트만으로 시작하라.
UI는 아직 건드리지 마라 — 다음 단계에서 할 것이다.

구현한 후 `npm test`와 `npm run build`를 실행하여
아무것도 망가지지 않았는지 확인하라."
```

각 단계에 무엇이 범위 내인지와 무엇이 범위 외인지를 명시적으로 말한다.

## Increment Checklist

각 단계 후 확인한다:

- [ ] 변경이 하나의 일을 완전히 한다
- [ ] 모든 기존 테스트가 여전히 통과한다 (`npm test`)
- [ ] 빌드가 성공한다 (`npm run build`)
- [ ] 타입 검사가 통과한다 (`npx tsc --noEmit`)
- [ ] 린팅이 통과한다 (`npm run lint`)
- [ ] 새 기능이 예상대로 작동한다
- [ ] 변경이 설명적인 메시지와 함께 커밋되었다

## Common Rationalizations

| 합리화 | 현실 |
|--------|------|
| "끝에 모두 테스트할 것이다" | 버그가 누적된다. 슬라이스 1의 버그는 슬라이스 2-5를 잘못되게 만든다. 각 슬라이스를 테스트하라. |
| "모두 한 번에 하는 것이 더 빠르다" | 500개의 변경된 줄 중 어떤 것이 원인인지 찾을 수 없을 때 무언가 망가질 때까지는 *느낌*이 빠르다. |
| "이 변경들은 너무 작아서 별도로 커밋할 수 없다" | 소규모 커밋은 무료다. 대규모 커밋은 버그를 숨기고 롤백을 고통스럽게 만든다. |
| "나중에 feature flag를 추가할 것이다" | 기능이 완성되지 않았다면, 사용자에게 보여지지 않아야 한다. 지금 flag를 추가하라. |
| "이 리팩터는 포함할 정도로 충분히 작다" | 기능과 혼합된 리팩터는 두 가지 모두 검토하고 디버깅하기 더 어렵게 만든다. 분리하라. |

## Red Flags

- 테스트 실행 없이 100줄 이상의 코드 작성
- 단일 단계에서 여러 무관한 변경
- "빠르게 이것도 추가하자" 범위 확장
- 더 빠르게 이동하기 위해 테스트/확인 단계 건너뛰기
- 단계 사이에 빌드 또는 테스트 망가짐
- 커밋되지 않은 대규모 변경이 쌓임
- 세 번째 사용 사례가 요구하기 전에 추상화 구축
- "이 작업을 하는 김에" 작업 범위 밖 파일 건드리기
- 일회성 작업을 위한 새 유틸리티 파일 생성

## Verification

작업에 대한 모든 단계 완료 후:

- [ ] 각 단계가 개별적으로 테스트되고 커밋되었다
- [ ] 전체 테스트 스위트가 통과한다
- [ ] 빌드가 깔끔하다
- [ ] 명세대로 기능이 end-to-end로 작동한다
- [ ] 커밋되지 않은 변경이 남아 있지 않다
