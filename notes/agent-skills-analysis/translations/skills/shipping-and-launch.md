# 배포와 런치

## 개요

자신감 있게 배포하세요. 목표는 단순히 배포하는 것이 아닙니다 — 모니터링을 갖추고, 롤백 계획을 준비하고, 성공이 어떤 모습인지 명확히 이해한 상태에서 안전하게 배포하는 것입니다. 모든 런치는 되돌릴 수 있고, 관찰 가능하며, 점진적이어야 합니다.

## 사용 시점

- 기능을 처음으로 프로덕션에 배포할 때
- 사용자에게 중요한 변경 사항을 릴리스할 때
- 데이터나 인프라를 마이그레이션할 때
- 베타 또는 얼리 액세스 프로그램을 시작할 때
- 위험을 수반하는 모든 배포 (모두 해당됨)

## 런치 전 체크리스트

### 코드 품질

- [ ] 모든 테스트 통과 (유닛, 통합, e2e)
- [ ] 경고 없이 빌드 성공
- [ ] 린트 및 타입 체크 통과
- [ ] 코드 검토 및 승인됨
- [ ] 런치 전 해결해야 할 TODO 주석 없음
- [ ] 프로덕션 코드에 `console.log` 디버깅 구문 없음
- [ ] 예상되는 실패 모드에 대한 오류 처리 완비

### 보안

- [ ] 코드나 버전 관리에 비밀 정보 없음
- [ ] `npm audit`에서 critical 또는 high 취약점 없음
- [ ] 모든 사용자 대면 엔드포인트에 입력 유효성 검증
- [ ] 인증 및 인가 확인 완비
- [ ] 보안 헤더 설정됨 (CSP, HSTS 등)
- [ ] 인증 엔드포인트에 속도 제한
- [ ] CORS가 특정 출처로 설정됨 (와일드카드 아님)

### 성능

- [ ] Core Web Vitals가 "좋음" 기준 이내
- [ ] 핵심 경로에 N+1 쿼리 없음
- [ ] 이미지 최적화됨 (압축, 반응형 크기, 지연 로딩)
- [ ] 번들 크기가 예산 이내
- [ ] 데이터베이스 쿼리에 적절한 인덱스 존재
- [ ] 정적 에셋 및 반복 쿼리에 캐싱 설정됨

### 접근성

- [ ] 모든 인터랙티브 요소에서 키보드 탐색 작동
- [ ] 스크린 리더가 페이지 콘텐츠와 구조를 전달할 수 있음
- [ ] 색상 대비가 WCAG 2.1 AA 기준 충족 (텍스트 4.5:1)
- [ ] 모달 및 동적 콘텐츠에 대한 포커스 관리 올바름
- [ ] 오류 메시지가 설명적이고 폼 필드와 연결됨
- [ ] axe-core 또는 Lighthouse에서 접근성 경고 없음

### 인프라

- [ ] 프로덕션에 환경 변수 설정됨
- [ ] 데이터베이스 마이그레이션 적용됨 (또는 적용 준비됨)
- [ ] DNS 및 SSL 설정됨
- [ ] 정적 에셋에 CDN 설정됨
- [ ] 로깅 및 오류 보고 설정됨
- [ ] 헬스 체크 엔드포인트가 존재하고 응답함

### 문서

- [ ] 새로운 설정 요구사항으로 README 업데이트됨
- [ ] API 문서 최신 상태
- [ ] 아키텍처 결정에 대한 ADR 작성됨
- [ ] 변경 로그 업데이트됨
- [ ] 사용자 대면 문서 업데이트됨 (해당되는 경우)

## Feature Flag 전략

배포와 릴리스를 분리하기 위해 feature flag 뒤에 배포합니다:

```typescript
// Feature flag 확인
const flags = await getFeatureFlags(userId);

if (flags.taskSharing) {
  // 새 기능: 태스크 공유
  return <TaskSharingPanel task={task} />;
}

// 기본값: 기존 동작
return null;
```

**Feature flag 생명주기:**

```
1. 플래그 OFF로 배포     → 코드가 프로덕션에 있지만 비활성 상태
2. 팀/베타에 활성화      → 프로덕션 환경에서 내부 테스트
3. 점진적 롤아웃         → 사용자의 5% → 25% → 50% → 100%
4. 각 단계에서 모니터링  → 오류율, 성능, 사용자 피드백 확인
5. 정리                  → 전체 롤아웃 후 플래그와 비활성 코드 경로 제거
```

**규칙:**
- 모든 feature flag에는 소유자와 만료 날짜가 있음
- 전체 롤아웃 후 2주 이내에 플래그 정리
- feature flag 중첩 금지 (지수적 조합 생성)
- CI에서 플래그 상태(켜짐과 꺼짐) 모두 테스트

## 단계적 롤아웃

### 롤아웃 순서

```
1. 스테이징에 배포
   └── 스테이징 환경에서 전체 테스트 스위트 실행
   └── 핵심 플로우 수동 스모크 테스트

2. 프로덕션에 배포 (feature flag OFF)
   └── 배포 성공 확인 (헬스 체크)
   └── 오류 모니터링 확인 (새 오류 없음)

3. 팀에 활성화 (내부 사용자에게 플래그 ON)
   └── 팀이 프로덕션에서 기능 사용
   └── 24시간 모니터링 기간

4. 카나리 롤아웃 (사용자 5%에게 플래그 ON)
   └── 오류율, 지연 시간, 사용자 행동 모니터링
   └── 메트릭 비교: 카나리 vs. 베이스라인
   └── 24-48시간 모니터링 기간
   └── 모든 임계값 통과 시에만 진행 (아래 표 참고)

5. 점진적 증가 (25% -> 50% -> 100%)
   └── 각 단계에서 동일한 모니터링
   └── 어느 시점에서든 이전 비율로 롤백 가능

6. 전체 롤아웃 (모든 사용자에게 플래그 ON)
   └── 1주일간 모니터링
   └── feature flag 정리
```

### 롤아웃 결정 임계값

각 단계에서 진행, 보류, 또는 롤백 여부를 결정하려면 이 임계값을 사용하세요:

| 메트릭 | 진행 (녹색) | 보류 및 조사 (노란색) | 롤백 (빨간색) |
|--------|------------|----------------------|--------------|
| 오류율 | 베이스라인의 10% 이내 | 베이스라인보다 10-100% 높음 | 베이스라인의 2배 초과 |
| P95 지연 시간 | 베이스라인의 20% 이내 | 베이스라인보다 20-50% 높음 | 베이스라인보다 50% 초과 |
| 클라이언트 JS 오류 | 새로운 오류 유형 없음 | 세션의 0.1% 미만에서 새 오류 | 세션의 0.1% 초과에서 새 오류 |
| 비즈니스 메트릭 | 중립 또는 긍정적 | 5% 미만 감소 (노이즈일 수 있음) | 5% 초과 감소 |

### 롤백 시점

다음의 경우 즉시 롤백합니다:
- 오류율이 베이스라인의 2배 이상 증가
- P95 지연 시간이 50% 이상 증가
- 사용자 보고 문제 급증
- 데이터 무결성 문제 감지
- 보안 취약점 발견

## 모니터링과 관찰 가능성

### 모니터링할 항목

```
애플리케이션 메트릭:
├── 오류율 (전체 및 엔드포인트별)
├── 응답 시간 (p50, p95, p99)
├── 요청 볼륨
├── 활성 사용자
└── 핵심 비즈니스 메트릭 (전환율, 참여도)

인프라 메트릭:
├── CPU 및 메모리 사용률
├── 데이터베이스 연결 풀 사용량
├── 디스크 공간
├── 네트워크 지연 시간
└── 큐 깊이 (해당되는 경우)

클라이언트 메트릭:
├── Core Web Vitals (LCP, INP, CLS)
├── JavaScript 오류
├── 클라이언트 관점에서의 API 오류율
└── 페이지 로드 시간
```

### 오류 보고

```typescript
// 보고 기능이 있는 오류 경계 설정
class ErrorBoundary extends React.Component {
  componentDidCatch(error: Error, info: React.ErrorInfo) {
    // 오류 추적 서비스에 보고
    reportError(error, {
      componentStack: info.componentStack,
      userId: getCurrentUser()?.id,
      page: window.location.pathname,
    });
  }

  render() {
    if (this.state.hasError) {
      return <ErrorFallback onRetry={() => this.setState({ hasError: false })} />;
    }
    return this.props.children;
  }
}

// 서버 측 오류 보고
app.use((err: Error, req: Request, res: Response, next: NextFunction) => {
  reportError(err, {
    method: req.method,
    url: req.url,
    userId: req.user?.id,
  });

  // 사용자에게 내부 정보 노출 금지
  res.status(500).json({
    error: { code: 'INTERNAL_ERROR', message: '문제가 발생했습니다' },
  });
});
```

### 런치 후 검증

런치 후 첫 한 시간 동안:

```
1. 헬스 엔드포인트가 200을 반환하는지 확인
2. 오류 모니터링 대시보드 확인 (새로운 오류 유형 없음)
3. 지연 시간 대시보드 확인 (회귀 없음)
4. 핵심 사용자 플로우 수동 테스트
5. 로그가 흐르고 읽을 수 있는지 확인
6. 롤백 메커니즘 작동 확인 (가능하면 드라이런)
```

## 롤백 전략

모든 배포는 실행 전에 롤백 계획이 필요합니다:

```markdown
## [기능/릴리스] 롤백 계획

### 트리거 조건
- 오류율 > 베이스라인의 2배
- P95 지연 시간 > [X]ms
- [특정 문제]에 대한 사용자 보고

### 롤백 단계
1. feature flag 비활성화 (해당되는 경우)
   또는
1. 이전 버전 배포: `git revert <commit> && git push`
2. 롤백 확인: 헬스 체크, 오류 모니터링
3. 소통: 팀에 롤백 알림

### 데이터베이스 고려사항
- 마이그레이션 [X]에 롤백 있음: `npx prisma migrate rollback`
- 새 기능으로 삽입된 데이터: [보존 / 정리]

### 롤백 소요 시간
- Feature flag: 1분 미만
- 이전 버전 재배포: 5분 미만
- 데이터베이스 롤백: 15분 미만
```

## 흔한 합리화

| 합리화 | 현실 |
|--------|------|
| "스테이징에서 동작하면 프로덕션에서도 동작할 거야" | 프로덕션은 데이터, 트래픽 패턴, 엣지 케이스가 다릅니다. 배포 후 모니터링하세요. |
| "이건 feature flag가 필요 없어" | 모든 기능은 킬 스위치의 혜택을 받습니다. "단순한" 변경도 문제를 일으킬 수 있습니다. |
| "모니터링은 오버헤드야" | 모니터링이 없다는 것은 대시보드 대신 사용자 불만을 통해 문제를 발견한다는 의미입니다. |
| "나중에 모니터링 추가할게" | 런치 전에 추가하세요. 볼 수 없는 것은 디버깅할 수 없습니다. |
| "롤백은 실패를 인정하는 것" | 롤백은 책임감 있는 엔지니어링입니다. 고장난 기능을 배포하는 것이 실패입니다. |

## 위험 신호

- 롤백 계획 없이 배포
- 프로덕션에 모니터링이나 오류 보고 없음
- 빅뱅 릴리스 (스테이징 없이 한 번에 모든 것)
- 만료 날짜나 소유자 없는 feature flag
- 배포 후 첫 한 시간 동안 아무도 모니터링하지 않음
- 프로덕션 환경 설정을 기억에 의존하여 수행 (코드가 아님)
- "금요일 오후인데, 그냥 배포하자"

## 검증

배포 전:

- [ ] 런치 전 체크리스트 완료 (모든 섹션 녹색)
- [ ] feature flag 설정됨 (해당되는 경우)
- [ ] 롤백 계획 문서화됨
- [ ] 모니터링 대시보드 설정됨
- [ ] 배포에 대해 팀에 알림

배포 후:

- [ ] 헬스 체크가 200 반환
- [ ] 오류율이 정상
- [ ] 지연 시간이 정상
- [ ] 핵심 사용자 플로우 동작
- [ ] 로그가 흐르고 있음
- [ ] 롤백 테스트됨 또는 준비 완료 확인됨
