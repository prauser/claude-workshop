# Source-Driven Development

## 개요

모든 프레임워크 관련 코드 결정은 공식 문서에 의해 뒷받침되어야 합니다. 기억에 의존해서 구현하지 마세요 — 검증하고, 출처를 인용하고, 사용자가 출처를 볼 수 있게 하세요. 훈련 데이터는 오래되고, API는 지원이 중단되며, 모범 사례는 진화합니다. 이 스킬은 모든 패턴이 사용자가 확인할 수 있는 권위 있는 출처로 추적되기 때문에 신뢰할 수 있는 코드를 사용자가 얻을 수 있도록 보장합니다.

## 사용 시점

- 사용자가 특정 프레임워크의 현재 모범 사례를 따르는 코드를 원할 때
- 프로젝트 전반에 걸쳐 복사될 보일러플레이트, 스타터 코드, 또는 패턴을 구축할 때
- 사용자가 명시적으로 문서화되고, 검증되었거나, "올바른" 구현을 요청할 때
- 프레임워크의 권장 방식이 중요한 기능을 구현할 때 (폼, 라우팅, 데이터 페칭, 상태 관리, 인증)
- 프레임워크 특정 패턴을 사용하는 코드를 검토하거나 개선할 때
- 기억에 의존해서 프레임워크 특정 코드를 작성하려 할 때

**사용하지 않을 때:**

- 특정 버전에 정확성이 의존하지 않을 때 (변수명 변경, 오타 수정, 파일 이동)
- 모든 버전에서 동일하게 작동하는 순수 로직 (루프, 조건문, 데이터 구조)
- 사용자가 명시적으로 검증보다 속도를 원할 때 ("빠르게 그냥 해줘")

## 프로세스

```
감지 ──→ 가져오기 ──→ 구현 ──→ 인용
 │          │           │          │
 ▼          ▼           ▼          ▼
어떤       관련         문서화된   출처
스택?      문서 가져옴  패턴 따름  제시
```

### 1단계: 스택과 버전 감지

프로젝트의 의존성 파일을 읽어 정확한 버전을 파악합니다:

```
package.json    → Node/React/Vue/Angular/Svelte
composer.json   → PHP/Symfony/Laravel
requirements.txt / pyproject.toml → Python/Django/Flask
go.mod          → Go
Cargo.toml      → Rust
Gemfile         → Ruby/Rails
```

발견한 내용을 명시적으로 기술합니다:

```
감지된 스택:
- React 19.1.0 (package.json에서)
- Vite 6.2.0
- Tailwind CSS 4.0.3
→ 관련 패턴에 대한 공식 문서를 가져오겠습니다.
```

버전이 없거나 모호하면 **사용자에게 물어보세요.** 추측하지 마세요 — 버전에 따라 올바른 패턴이 결정됩니다.

### 2단계: 공식 문서 가져오기

구현하려는 기능에 대한 특정 문서 페이지를 가져옵니다. 홈페이지나 전체 문서가 아닌 — 관련 페이지를 가져옵니다.

**출처 우선순위 (권위 순서):**

| 우선순위 | 출처 | 예시 |
|---------|------|------|
| 1 | 공식 문서 | react.dev, docs.djangoproject.com, symfony.com/doc |
| 2 | 공식 블로그 / 변경 로그 | react.dev/blog, nextjs.org/blog |
| 3 | 웹 표준 참고 자료 | MDN, web.dev, html.spec.whatwg.org |
| 4 | 브라우저/런타임 호환성 | caniuse.com, node.green |

**권위 있지 않음 — 절대 주요 출처로 인용하지 말 것:**

- Stack Overflow 답변
- 블로그 포스트 또는 튜토리얼 (인기 있는 것도 포함)
- AI가 생성한 문서나 요약
- 본인의 훈련 데이터 (바로 그것이 요점입니다 — 검증하세요)

**가져오는 것에 정확해야 합니다:**

```
나쁜 예:  React 홈페이지를 가져옴
좋은 예: react.dev/reference/react/useActionState를 가져옴

나쁜 예:  "django 인증 모범 사례" 검색
좋은 예: docs.djangoproject.com/en/6.0/topics/auth/ 를 가져옴
```

가져온 후, 핵심 패턴을 추출하고 지원 중단 경고나 마이그레이션 안내를 메모합니다.

공식 출처들이 서로 충돌할 경우 (예: 마이그레이션 가이드가 API 레퍼런스와 모순될 때), 불일치를 사용자에게 알리고 감지된 버전에 실제로 어떤 패턴이 작동하는지 확인하세요.

### 3단계: 문서화된 패턴을 따라 구현

문서가 보여주는 것과 일치하는 코드를 작성합니다:

- 기억이 아닌 문서의 API 시그니처를 사용
- 문서가 새로운 방법을 보여주면 새로운 방법을 사용
- 문서가 패턴을 지원 중단으로 표시하면 지원 중단된 버전을 사용하지 말 것
- 문서가 무언가를 다루지 않으면 미검증으로 표시

**문서가 기존 프로젝트 코드와 충돌할 때:**

```
충돌 감지:
기존 코드베이스는 폼 로딩 상태에 useState를 사용하지만,
React 19 문서는 이 패턴에 useActionState를 권장합니다.
(출처: react.dev/reference/react/useActionState)

옵션:
A) 현대적인 패턴 사용 (useActionState) — 현재 문서와 일관성
B) 기존 코드와 일치 (useState) — 코드베이스와 일관성
→ 어떤 방식을 선호하시나요?
```

충돌을 드러내세요. 조용히 하나를 선택하지 마세요.

### 4단계: 출처 인용

모든 프레임워크 특정 패턴은 인용을 받습니다. 사용자는 모든 결정을 검증할 수 있어야 합니다.

**코드 주석에서:**

```typescript
// React 19 폼 처리, useActionState 사용
// 출처: https://react.dev/reference/react/useActionState#usage
const [state, formAction, isPending] = useActionState(submitOrder, initialState);
```

**대화에서:**

```
폼 제출 상태에 수동 useState 대신 useActionState를 사용하고 있습니다.
React 19는 이 훅으로 수동 isPending/setIsPending 패턴을 대체했습니다.

출처: https://react.dev/blog/2024/12/05/react-19#actions
"useTransition은 이제 비동기 함수를 지원합니다 [...] 대기 상태를 자동으로 처리하기 위해"
```

**인용 규칙:**

- 단축 URL이 아닌 전체 URL
- 가능하면 앵커가 있는 딥 링크를 선호합니다 (예: 최상위 페이지보다 `/useActionState#usage`) — 앵커는 문서 재구성 후에도 최상위 페이지보다 더 잘 살아남습니다
- 명백하지 않은 결정을 뒷받침할 때 관련 구절을 인용
- 플랫폼 기능 권장 시 브라우저/런타임 지원 데이터 포함
- 패턴에 대한 문서를 찾을 수 없으면 명시적으로 말합니다:

```
미검증: 이 패턴에 대한 공식 문서를 찾을 수 없었습니다.
이것은 훈련 데이터를 기반으로 하며 구식일 수 있습니다.
프로덕션에서 사용하기 전에 검증하세요.
```

검증할 수 없었던 것에 대한 솔직함이 거짓 확신보다 더 가치 있습니다.

## 흔한 합리화

| 합리화 | 현실 |
|--------|------|
| "이 API에 대해 확신해" | 확신은 증거가 아닙니다. 훈련 데이터에는 올바르게 보이지만 현재 버전에서 실패하는 구식 패턴이 포함되어 있습니다. 검증하세요. |
| "문서 가져오면 토큰 낭비야" | API를 환각하는 것이 더 낭비입니다. 사용자는 한 시간 동안 디버깅한 다음 함수 시그니처가 변경되었다는 것을 발견합니다. 한 번의 가져오기가 수 시간의 재작업을 방지합니다. |
| "문서에 필요한 게 없을 거야" | 문서가 다루지 않는다면 그것은 가치 있는 정보입니다 — 패턴이 공식적으로 권장되지 않을 수 있습니다. |
| "구식일 수 있다고 언급할게" | 면책 조항은 도움이 되지 않습니다. 검증하고 인용하거나, 명확하게 미검증으로 표시하세요. 애매하게 처리하는 것이 최악의 옵션입니다. |
| "단순한 작업이라서 확인할 필요 없어" | 잘못된 패턴이 있는 단순한 작업은 템플릿이 됩니다. 사용자는 현대적인 방식이 존재한다는 것을 발견하기 전에 지원 중단된 폼 핸들러를 열 개의 컴포넌트에 복사합니다. |

## 위험 신호

- 해당 버전의 문서를 확인하지 않고 프레임워크 특정 코드 작성
- 출처를 인용하는 대신 API에 대해 "~인 것 같아" 또는 "~라고 생각해" 사용
- 어떤 버전에 적용되는지 모르는 패턴 구현
- 공식 문서 대신 Stack Overflow나 블로그 포스트 인용
- 훈련 데이터에 나타나기 때문에 지원 중단된 API 사용
- 구현 전 `package.json` / 의존성 파일을 읽지 않음
- 프레임워크 특정 결정에 대한 출처 인용 없이 코드 제공
- 한 페이지만 관련 있을 때 전체 문서 사이트 가져오기

## 검증

Source-driven development로 구현한 후:

- [ ] 의존성 파일에서 프레임워크와 라이브러리 버전이 파악됨
- [ ] 프레임워크 특정 패턴에 대해 공식 문서가 가져와짐
- [ ] 모든 출처가 공식 문서이며 블로그 포스트나 훈련 데이터가 아님
- [ ] 코드가 현재 버전 문서에 표시된 패턴을 따름
- [ ] 비자명한 결정에 전체 URL이 포함된 출처 인용이 있음
- [ ] 지원 중단된 API가 사용되지 않음 (마이그레이션 가이드 대조 확인)
- [ ] 문서와 기존 코드 간의 충돌이 사용자에게 드러남
- [ ] 검증할 수 없었던 모든 것이 명시적으로 미검증으로 표시됨
