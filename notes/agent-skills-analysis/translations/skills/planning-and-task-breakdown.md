# 계획 수립과 태스크 분해

## 개요

작업을 명시적인 완료 기준이 있는 작고 검증 가능한 태스크로 분해합니다. 좋은 태스크 분해는 신뢰성 있게 작업을 완료하는 에이전트와 뒤엉킨 혼란을 만들어내는 에이전트의 차이를 결정합니다. 모든 태스크는 단 한 번의 집중된 세션 안에서 구현, 테스트, 검증할 수 있을 만큼 작아야 합니다.

## 사용 시점

- 스펙이 있고 구현 가능한 단위로 나눠야 할 때
- 태스크가 너무 크거나 모호하게 느껴져 시작하기 어려울 때
- 여러 에이전트나 세션에 걸쳐 작업을 병렬화해야 할 때
- 사람에게 범위를 전달해야 할 때
- 구현 순서가 명확하지 않을 때

**사용하지 않을 때:** 단일 파일 변경으로 범위가 명확하거나, 스펙에 이미 잘 정의된 태스크가 포함된 경우.

## 계획 수립 프로세스

### 1단계: 계획 모드 진입

코드를 작성하기 전에 읽기 전용 모드로 운영합니다:

- 스펙과 관련 코드베이스 섹션을 읽습니다
- 기존 패턴과 관례를 파악합니다
- 컴포넌트 간 의존성을 매핑합니다
- 위험 요소와 불확실한 부분을 메모합니다

**계획 단계에서는 코드를 작성하지 마세요.** 산출물은 계획 문서이지 구현이 아닙니다.

### 2단계: 의존성 그래프 파악

무엇이 무엇에 의존하는지 매핑합니다:

```
데이터베이스 스키마
    │
    ├── API 모델/타입
    │       │
    │       ├── API 엔드포인트
    │       │       │
    │       │       └── 프론트엔드 API 클라이언트
    │       │               │
    │       │               └── UI 컴포넌트
    │       │
    │       └── 유효성 검증 로직
    │
    └── 시드 데이터 / 마이그레이션
```

구현 순서는 의존성 그래프의 아래에서 위 방향을 따릅니다: 기반을 먼저 구축합니다.

### 3단계: 수직으로 슬라이싱하기

데이터베이스 전체를 먼저 구축하고, 그다음 API 전체, 그다음 UI 전체를 구축하는 대신 — 한 번에 하나의 완전한 기능 경로를 구축합니다:

**나쁜 예 (수평 슬라이싱):**
```
태스크 1: 전체 데이터베이스 스키마 구축
태스크 2: 모든 API 엔드포인트 구축
태스크 3: 모든 UI 컴포넌트 구축
태스크 4: 전체 연결
```

**좋은 예 (vertical slicing):**
```
태스크 1: 사용자가 계정을 생성할 수 있음 (회원가입을 위한 스키마 + API + UI)
태스크 2: 사용자가 로그인할 수 있음 (인증 스키마 + API + 로그인 UI)
태스크 3: 사용자가 태스크를 생성할 수 있음 (태스크 스키마 + API + 생성 UI)
태스크 4: 사용자가 태스크 목록을 볼 수 있음 (쿼리 + API + 목록 보기 UI)
```

각 vertical slice는 동작하고 테스트 가능한 기능을 제공합니다.

### 4단계: 태스크 작성

각 태스크는 다음 구조를 따릅니다:

```markdown
## 태스크 [N]: [짧고 설명적인 제목]

**설명:** 이 태스크가 달성하는 내용을 설명하는 한 문단.

**완료 기준:**
- [ ] [구체적이고 테스트 가능한 조건]
- [ ] [구체적이고 테스트 가능한 조건]

**검증:**
- [ ] 테스트 통과: `npm test -- --grep "feature-name"`
- [ ] 빌드 성공: `npm run build`
- [ ] 수동 확인: [검증할 내용 설명]

**의존성:** [이 태스크가 의존하는 태스크 번호, 또는 "없음"]

**수정될 가능성이 있는 파일:**
- `src/path/to/file.ts`
- `tests/path/to/test.ts`

**예상 범위:** [소형: 파일 1-2개 | 중형: 파일 3-5개 | 대형: 파일 5개 이상]
```

### 5단계: 순서 정렬 및 체크포인트 설정

태스크를 다음과 같이 배열합니다:

1. 의존성이 충족됨 (기반 먼저 구축)
2. 각 태스크 완료 후 시스템이 동작하는 상태를 유지
3. 매 2-3개 태스크마다 검증 체크포인트가 존재
4. 위험도 높은 태스크를 앞에 배치 (빠른 실패 확인)

명시적인 체크포인트를 추가합니다:

```markdown
## 체크포인트: 태스크 1-3 완료 후
- [ ] 모든 테스트 통과
- [ ] 애플리케이션이 오류 없이 빌드됨
- [ ] 핵심 사용자 흐름이 end-to-end로 동작
- [ ] 계속 진행 전 사람이 검토
```

## 태스크 크기 가이드라인

| 크기 | 파일 수 | 범위 | 예시 |
|------|---------|------|------|
| **XS** | 1 | 단일 함수 또는 설정 변경 | 유효성 검증 규칙 추가 |
| **S** | 1-2 | 컴포넌트 하나 또는 엔드포인트 하나 | 새 API 엔드포인트 추가 |
| **M** | 3-5 | 기능 슬라이스 하나 | 사용자 회원가입 플로우 |
| **L** | 5-8 | 다중 컴포넌트 기능 | 필터링 및 페이지네이션이 있는 검색 |
| **XL** | 8개 이상 | **너무 큼 — 더 작게 분해할 것** | — |

태스크가 L 이상이라면 더 작은 태스크로 분해해야 합니다. 에이전트는 S와 M 크기의 태스크에서 가장 잘 수행합니다.

**태스크를 더 분해해야 할 때:**
- 한 번의 집중 세션(에이전트 작업 기준 약 2시간 이상)보다 오래 걸릴 것 같을 때
- 완료 기준을 3개 이하의 항목으로 설명할 수 없을 때
- 두 개 이상의 독립적인 서브시스템(예: 인증과 결제)을 건드릴 때
- 태스크 제목에 "그리고"를 쓰고 있는 자신을 발견할 때 (두 개의 태스크라는 신호)

## 계획 문서 템플릿

```markdown
# 구현 계획: [기능/프로젝트 이름]

## 개요
[구축하는 내용에 대한 한 문단 요약]

## 아키텍처 결정
- [핵심 결정 1과 근거]
- [핵심 결정 2와 근거]

## 태스크 목록

### 1단계: 기반 구축
- [ ] 태스크 1: ...
- [ ] 태스크 2: ...

### 체크포인트: 기반
- [ ] 테스트 통과, 빌드 정상

### 2단계: 핵심 기능
- [ ] 태스크 3: ...
- [ ] 태스크 4: ...

### 체크포인트: 핵심 기능
- [ ] End-to-end 플로우 동작

### 3단계: 마무리
- [ ] 태스크 5: ...
- [ ] 태스크 6: ...

### 체크포인트: 완료
- [ ] 모든 완료 기준 충족
- [ ] 검토 준비 완료

## 위험 요소와 대응 방안
| 위험 요소 | 영향도 | 대응 방안 |
|-----------|--------|-----------|
| [위험 요소] | [상/중/하] | [전략] |

## 미해결 질문
- [사람의 입력이 필요한 질문]
```

## 병렬화 기회

여러 에이전트나 세션을 사용할 수 있을 때:

- **병렬화 가능:** 독립적인 기능 슬라이스, 이미 구현된 기능에 대한 테스트, 문서화
- **순차 처리 필수:** 데이터베이스 마이그레이션, 공유 상태 변경, 의존성 체인
- **조율 필요:** API 계약을 공유하는 기능 (계약을 먼저 정의한 후 병렬화)

## 흔한 합리화

| 합리화 | 현실 |
|--------|------|
| "진행하면서 파악할 거야" | 그것이 뒤엉킨 혼란과 재작업으로 끝나는 방법입니다. 10분의 계획이 수 시간을 절약합니다. |
| "태스크는 명확해" | 그래도 적어두세요. 명시적인 태스크는 숨겨진 의존성과 잊혀진 엣지 케이스를 드러냅니다. |
| "계획은 오버헤드야" | 계획이 바로 태스크입니다. 계획 없는 구현은 그저 타이핑일 뿐입니다. |
| "다 머릿속에 있어" | 컨텍스트 창은 유한합니다. 문서화된 계획은 세션 경계와 컴팩션을 넘어 살아남습니다. |

## 위험 신호

- 작성된 태스크 목록 없이 구현 시작
- "기능 구현"이라고 쓰여 있지만 완료 기준이 없는 태스크
- 계획에 검증 단계가 없음
- 모든 태스크가 XL 크기
- 태스크 간 체크포인트 없음
- 의존성 순서를 고려하지 않음

## 검증

구현 시작 전 다음을 확인합니다:

- [ ] 모든 태스크에 완료 기준이 있음
- [ ] 모든 태스크에 검증 단계가 있음
- [ ] 태스크 의존성이 파악되고 올바르게 순서가 정해져 있음
- [ ] 어떤 태스크도 약 5개 이상의 파일을 건드리지 않음
- [ ] 주요 단계 사이에 체크포인트가 존재
- [ ] 사람이 계획을 검토하고 승인함
