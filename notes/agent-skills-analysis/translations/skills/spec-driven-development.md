# Spec-Driven Development

## 개요

코드를 작성하기 전에 구조화된 스펙을 작성합니다. 스펙은 여러분과 사람 엔지니어 사이의 공유된 진실의 원천입니다 — 우리가 무엇을 구축하고 있는지, 왜 구축하는지, 어떻게 완료되었다는 것을 알 수 있는지를 정의합니다. 스펙 없는 코드는 추측입니다.

## 사용 시점

- 새 프로젝트나 기능을 시작할 때
- 요구사항이 모호하거나 불완전할 때
- 변경 사항이 여러 파일이나 모듈에 영향을 미칠 때
- 아키텍처 결정을 내리려 할 때
- 구현하는 데 30분 이상 걸리는 태스크

**사용하지 않을 때:** 한 줄 수정, 오타 교정, 또는 요구사항이 명확하고 독립적인 변경.

## 게이트 워크플로우

Spec-driven development에는 네 가지 단계가 있습니다. 현재 단계가 검증되기 전까지 다음 단계로 넘어가지 마세요.

```
스펙 작성 ──→ 계획 ──→ 태스크 ──→ 구현
   │           │        │          │
   ▼           ▼        ▼          ▼
  사람        사람      사람       사람
  검토        검토      검토       검토
```

### 1단계: 스펙 작성

높은 수준의 비전으로 시작합니다. 요구사항이 구체적이 될 때까지 사람에게 명확화 질문을 합니다.

**즉시 가정 사항을 드러냅니다.** 스펙 내용을 작성하기 전에 가정하고 있는 것들을 나열합니다:

```
가정하고 있는 사항:
1. 웹 애플리케이션입니다 (네이티브 모바일이 아님)
2. 인증은 세션 기반 쿠키를 사용합니다 (JWT가 아님)
3. 데이터베이스는 PostgreSQL입니다 (기존 Prisma 스키마 기반)
4. 최신 브라우저만 대상으로 합니다 (IE11 없음)
→ 지금 수정해주시지 않으면 이 사항들로 진행하겠습니다.
```

모호한 요구사항을 조용히 채워넣지 마세요. 스펙의 전체 목적은 코드가 작성되기 *전에* 오해를 드러내는 것입니다 — 가정은 가장 위험한 형태의 오해입니다.

**이 여섯 가지 핵심 영역을 다루는 스펙 문서를 작성합니다:**

1. **목표** — 무엇을 구축하고 왜 구축하는가? 사용자는 누구인가? 성공은 어떤 모습인가?

2. **명령어** — 도구 이름만이 아닌 플래그가 포함된 완전히 실행 가능한 명령어.
   ```
   빌드: npm run build
   테스트: npm test -- --coverage
   린트: npm run lint --fix
   개발: npm run dev
   ```

3. **프로젝트 구조** — 소스 코드 위치, 테스트 위치, 문서 위치.
   ```
   src/           → 애플리케이션 소스 코드
   src/components → React 컴포넌트
   src/lib        → 공유 유틸리티
   tests/         → 유닛 및 통합 테스트
   e2e/           → End-to-end 테스트
   docs/          → 문서
   ```

4. **코드 스타일** — 스타일을 설명하는 세 문단보다 실제 코드 스니펫 하나가 더 낫습니다. 네이밍 규칙, 포매팅 규칙, 좋은 출력의 예시를 포함합니다.

5. **테스트 전략** — 어떤 프레임워크, 테스트 위치, 커버리지 기대치, 어떤 우려사항에 어떤 테스트 레벨.

6. **경계** — 3단계 시스템:
   - **항상 수행:** 커밋 전 테스트 실행, 네이밍 규칙 따르기, 입력 유효성 검증
   - **먼저 확인:** 데이터베이스 스키마 변경, 의존성 추가, CI 설정 변경
   - **절대 안 됨:** 비밀 정보 커밋, 벤더 디렉토리 편집, 승인 없이 실패하는 테스트 제거

**스펙 템플릿:**

```markdown
# 스펙: [프로젝트/기능 이름]

## 목표
[구축하는 것과 이유. 사용자 스토리 또는 완료 기준.]

## 기술 스택
[프레임워크, 언어, 버전이 포함된 핵심 의존성]

## 명령어
[빌드, 테스트, 린트, 개발 — 전체 명령어]

## 프로젝트 구조
[설명이 있는 디렉토리 레이아웃]

## 코드 스타일
[예시 스니펫 + 핵심 규칙]

## 테스트 전략
[프레임워크, 테스트 위치, 커버리지 요구사항, 테스트 레벨]

## 경계
- 항상: [...]
- 먼저 확인: [...]
- 절대 안 됨: [...]

## 성공 기준
[완료됐다는 것을 알 수 있는 방법 — 구체적이고 테스트 가능한 조건]

## 미해결 질문
[사람의 입력이 필요한 미해결 사항]
```

**지시사항을 성공 기준으로 재구성합니다.** 모호한 요구사항을 받으면 구체적인 조건으로 번역합니다:

```
요구사항: "대시보드를 더 빠르게 만들어"

재구성된 성공 기준:
- 4G 연결에서 대시보드 LCP < 2.5s
- 초기 데이터 로드 < 500ms 완료
- 로드 중 레이아웃 이동 없음 (CLS < 0.1)
→ 이것이 올바른 목표인가요?
```

이를 통해 "더 빠르다"가 무엇을 의미하는지 추측하는 대신 명확한 목표를 향해 루프, 재시도, 문제 해결을 할 수 있습니다.

### 2단계: 계획

검증된 스펙으로 기술적 구현 계획을 생성합니다:

1. 주요 컴포넌트와 의존성 파악
2. 구현 순서 결정 (무엇을 먼저 구축해야 하는지)
3. 위험 요소와 완화 전략 메모
4. 병렬로 구축 가능한 것과 순차적으로 해야 하는 것 파악
5. 단계 사이의 검증 체크포인트 정의

계획은 검토 가능해야 합니다: 사람이 읽고 "예, 그게 올바른 방법이에요" 또는 "아니요, X를 바꾸세요"라고 말할 수 있어야 합니다.

### 3단계: 태스크

계획을 개별적으로 구현 가능한 태스크로 분해합니다:

- 각 태스크는 한 번의 집중된 세션에서 완료 가능
- 각 태스크에는 명시적인 완료 기준이 있음
- 각 태스크에는 검증 단계가 포함됨 (테스트, 빌드, 수동 확인)
- 태스크는 중요도가 아닌 의존성 순서로 정렬됨
- 어떤 태스크도 약 5개 이상의 파일을 변경하지 않아야 함

**태스크 템플릿:**
```markdown
- [ ] 태스크: [설명]
  - 완료 기준: [완료 시 충족되어야 할 것]
  - 검증: [확인 방법 — 테스트 명령어, 빌드, 수동 확인]
  - 파일: [수정될 파일들]
```

### 4단계: 구현

`incremental-implementation`과 `test-driven-development` 스킬을 따라 태스크를 하나씩 실행합니다. `context-engineering`을 사용하여 각 단계에서 전체 스펙으로 에이전트를 덮치는 대신 올바른 스펙 섹션과 소스 파일을 로드합니다.

## 스펙을 살아있게 유지하기

스펙은 일회성 산출물이 아닌 살아있는 문서입니다:

- **결정이 바뀌면 업데이트** — 데이터 모델을 변경해야 한다는 것을 발견하면, 먼저 스펙을 업데이트한 다음 구현합니다.
- **범위가 변경되면 업데이트** — 추가되거나 제외된 기능은 스펙에 반영되어야 합니다.
- **스펙을 커밋** — 스펙은 코드와 함께 버전 관리에 속합니다.
- **PR에서 스펙 참조** — 각 PR이 구현하는 스펙 섹션에 링크를 달아주세요.

## 흔한 합리화

| 합리화 | 현실 |
|--------|------|
| "이건 단순해서 스펙이 필요 없어" | 단순한 태스크도 긴 스펙은 필요하지 않지만, 여전히 완료 기준은 필요합니다. 두 줄짜리 스펙도 괜찮습니다. |
| "코딩 후에 스펙을 쓸게" | 그것은 문서화이지 스펙이 아닙니다. 스펙의 가치는 코드 *전에* 명확성을 강제하는 데 있습니다. |
| "스펙이 우리를 느리게 할 거야" | 15분의 스펙이 수 시간의 재작업을 방지합니다. 15분의 폭포수 방식이 15시간의 디버깅보다 낫습니다. |
| "어차피 요구사항이 바뀔 거야" | 그래서 스펙이 살아있는 문서입니다. 구식 스펙도 스펙이 없는 것보다 낫습니다. |
| "사용자가 원하는 게 뭔지 알아" | 명확한 요청에도 암묵적인 가정이 있습니다. 스펙은 그 가정들을 드러냅니다. |

## 위험 신호

- 작성된 요구사항 없이 코드 작성 시작
- "그냥 구축 시작할까요?"라고 "완료"가 무엇을 의미하는지 명확히 하기 전에 묻기
- 어떤 스펙이나 태스크 목록에도 언급되지 않은 기능 구현
- 아키텍처 결정을 문서화하지 않고 내림
- "무엇을 구축할지 명확해"라서 스펙 건너뜀

## 검증

구현으로 진행하기 전에 다음을 확인합니다:

- [ ] 스펙이 여섯 가지 핵심 영역을 모두 다룸
- [ ] 사람이 스펙을 검토하고 승인함
- [ ] 성공 기준이 구체적이고 테스트 가능함
- [ ] 경계 (항상/먼저 확인/절대 안 됨)가 정의됨
- [ ] 스펙이 저장소의 파일로 저장됨
